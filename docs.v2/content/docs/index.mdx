PyMediate is a small library for Python 3.12+ that gives your application one type-safe way to ask for things. You describe what you want as a plain object, hand it to a single dispatcher, and get back a fully typed response:

```python
response = mediator.send(CreateUser(username="alice", email="alice@example.com"))
# response is UserCreated — inferred from the request type, checked by mypy
```

If you've never met the mediator pattern before, that line probably raises more questions than it answers. Start with the problem it solves.

## The problem

In every application there's code that *wants* something done and code that *knows how* to do it. The usual way to connect the two is a direct call:

```python
from users.service import UserService

def signup(form):
    service = UserService(db, mailer)
    return service.create_user(form.username, form.email)
```

This is fine at first, and for small codebases it stays fine. As an application grows, the direct call quietly accumulates costs:

- **Every caller knows the implementation.** The concrete class, its constructor, its dependencies, its module path — change any of them and every call site changes too.
- **Cross-cutting concerns have nowhere to live.** Logging, validation, and transactions get copy-pasted into each service method, or stack up as decorators that each caller has to remember.
- **The same operation gets wired up many ways.** `create_user` ends up called from a web view, a CLI command, and a background job, each constructing the service slightly differently — and each needing its own mocks in tests.

## The mediator pattern

The [mediator pattern](https://en.wikipedia.org/wiki/Mediator_pattern) removes the direct call. Instead of invoking business logic, you describe what you want as a plain message — a **request** — and hand it to a single dispatcher — the **mediator** — which routes it to the one place that knows how to do it — a **handler**:

```python
from dataclasses import dataclass
from pymediate import Request, Handler

@dataclass
class CreateUser(Request[UserCreated]):   # the message, declaring its response type
    username: str
    email: str

class CreateUserHandler(Handler[CreateUser]):   # the one place it's handled
    def __call__(self, request: CreateUser) -> UserCreated:
        ...

user = mediator.send(CreateUser(username="alice", email="alice@example.com"))
```

Now the caller knows only the message it sends and the mediator it sends it to. The handler knows only the request it receives. The web view, the CLI command, and the background job all send the same `CreateUser` and none of them can tell — or need to know — how it's fulfilled. And cross-cutting concerns get a real home: a **pipeline behavior** wraps every dispatch, so logging or validation is written once instead of scattered across handlers.

## Why PyMediate?

PyMediate implements the pattern entirely inside Python's type system — built on [PEP 695](https://peps.python.org/pep-0695/) generics, new in Python 3.12 — so the indirection costs you nothing you'd have with a direct call:

- **The link is a type.** `Handler[CreateUser]` declares what a handler handles, and dispatch is keyed on the class itself. Rename the request and every handler follows in the refactor.
- **You always know what comes back.** A request inherits `Request[UserCreated]`, and from then on `mediator.send()` returns `UserCreated` everywhere — inferred by mypy, autocompleted by your editor, no casts. The response can be any type at all: no framework base class, no serialization methods.
- **Wiring mistakes surface before requests flow.** Handler signatures are validated the moment the class is defined — a wrong parameter type, a missing return type, a sync/async mixup all raise at import. A second handler for the same request raises immediately, pointing at the file and line of the first.
- **Sync and async are equals.** `pymediate.aio` mirrors the sync API class for class — switch the import, add `await`, done.
- **It travels light.** Zero runtime dependencies, ships `py.typed`, and is checked under `mypy --strict`. `dependency-injector` support is an optional extra.

Evaluating it against other libraries? [How it compares](/docs/comparison) has a feature-by-feature look at the Python mediator ecosystem and a reproducible benchmark of what `send()` costs over a direct call.

## The moving parts

| Concept | Role |
| --- | --- |
| `Request[T]` | A message declaring what it wants and the response type `T` it expects |
| `Handler[R]` | The single place where a request type `R` gets handled |
| `Mediator` | Routes each request to its handler; the only thing callers depend on |
| `PipelineBehavior` | Optional middleware wrapping handlers — logging, validation, transactions |
| `Services` / `ServiceProvider` | Registration and resolution of handlers, behaviors, and their dependencies |
