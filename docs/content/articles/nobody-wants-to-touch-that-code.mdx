I've watched the same scene play out on more than one team.

Someone picks up an improvement everyone wants. It looks like two days. They start work and discover that the first change needs another change underneath it. That one needs a third. Soon the diff has crossed into code they don't understand, tests they don't trust, and configuration nobody has touched since the person who wrote it left.

The original change is still reasonable. The path to it isn't.

So the branch goes stale. The ticket returns to the backlog with a note saying it needs more design work. Six months later, somebody proposes the same improvement. Someone who remembers says: careful, we tried that.

That's how a codebase becomes untouchable. It rarely looks terrible. The individual functions may be clean. The problem is how much unrelated knowledge a person must gather before changing one of them.

It did not begin that way.

## Where it starts

Here is the first endpoint in a small shop:

```python title="app.py"
from flask import Flask, jsonify, request

from .db import session
from .models import Order

app = Flask(__name__)

@app.post("/orders")
def place_order():
    order = Order(items=request.json["items"])
    session.add(order)
    session.commit()
    return jsonify(id=order.id), 201
```

**This code is right.** The route receives input, does the work, and returns a response. A new developer can understand the feature without first learning how the whole application is arranged.

Then the shop grows. Placing an order starts checking stock, taking payment, and sending a confirmation email. Refunds arrive. Customers ask for invoices and exports. Keeping all of that inside the routes makes them harder to scan, so the order logic moves into a module:

```python title="orders/service.py"
from db import session
from inventory import stock_levels
from mail import mailer
from payments import payment_client

def place_order(items, customer_id): ...
def cancel_order(order_id): ...
def refund_order(order_id, amount): ...
```

The routes become small again. Several routes can share the same rules. The module gives the order feature a clear home.

More work collects there:

```python title="orders/service.py — later"
from audit import audit_log
from currency import exchange_rates
from db import session
from documents import pdf_renderer
from inventory import stock_levels
from mail import mailer
from payments import payment_client

def place_order(items, customer_id): ...
def cancel_order(order_id): ...
def refund_order(order_id, amount): ...
def invoice_pdf(order_id): ...
def monthly_statement(customer_id): ...
def export_orders(customer_id, fmt="csv"): ...
```

Every import supports a real requirement. Every function concerns orders. The file still has a clear theme.

But the functions do different work. Select one below and look at which imports it uses:

<BlastRadiusDiagram />

`export_orders()` only uses the database session. `invoice_pdf()` needs the session and PDF renderer. Placing an order needs stock, payment, mail, and the session.

Python still creates the module as one object. The first time anything imports `orders.service`, Python runs every import at the top. Asking for the export brings the payment client and mailer into the process even though the export never calls them.

That may be harmless for years. It becomes noticeable when one operation needs to go somewhere the rest of the module does not.

## The part that needs to move

One customer now has enough orders for the export endpoint to time out. The team decides to run it in a background worker and return immediately from HTTP.

The export rules are unchanged. Read the customer's order history, render a file, store it, and return a location. Only the place where the code runs is changing.

The worker imports `export_orders`. That loads the payment client, mailer, PDF renderer, stock service, exchange rates, and audit log. Some read web configuration while they load. The worker has no web application, so code unrelated to exporting can stop the import before the export runs.

The first improvement is simply to give the export its own module:

```python title="orders/export.py"
from dataclasses import dataclass

from db import session
from files import file_store

@dataclass
class ExportOrders:
    customer_id: int
    fmt: str = "csv"

@dataclass
class ExportResult:
    url: str
    rows: int

def export_orders(request: ExportOrders) -> ExportResult:
    rows = session.orders_for(request.customer_id)
    content = render(rows, request.fmt)
    url = file_store.write(content)
    return ExportResult(url=url, rows=len(rows))
```

This is still a module and a function. It still imports real implementations. What changed is its boundary.

The export now travels with the two things it uses rather than every dependency of the order feature. It accepts a customer ID instead of reaching into `g.current_user`, so the caller supplies the identity in whatever form it received it.

The HTTP route translates an HTTP request:

```python title="app.py"
@app.post("/orders/export")
def export_orders_route():
    export = ExportOrders(customer_id=g.current_user.id)
    result = export_orders(export)
    return jsonify(url=result.url), 202
```

The worker translates a queue message:

```python title="worker.py"
def handle_message(message):
    export_orders(ExportOrders(**message.body))
```

Flask stays in the route. The queue library stays in the worker. The export input belongs to neither, so both callers can construct it.

Nothing here depends on the module being tiny for its own sake. The useful boundary comes from keeping together the code that performs one business operation and excluding code that belongs to other operations.

The worker can now import the export without taking payment and email code with it. Whether it can run the function depends on the two implementations still imported by `orders/export.py`. That is the next problem.

## The thing that needs to change

The new worker needs a database connection with a different lifetime from the web request. Later, the shop may move exports to a different storage provider. In both cases, `orders/export.py` imports a particular implementation before the caller gets a say.

We could pass the database session and file store into the function on every call:

```python
result = export_orders(export, session, file_store)
```

That makes them replaceable, but it also makes every caller responsible for carrying them. The export always needs the same two objects, so we can supply them once:

```python title="orders/export.py"
class OrderExporter:
    def __init__(self, history, destination):
        self._history = history
        self._destination = destination

    def __call__(self, request: ExportOrders) -> ExportResult:
        rows = self._history.for_customer(request.customer_id)
        content = render(rows, request.fmt)
        url = self._destination.write(content)
        return ExportResult(url=url, rows=len(rows))
```

The class stores the two objects between calls. That is the job it adds. Startup constructs the class; routes and workers only provide `ExportOrders`.

There is one more useful step. The export does not need every method on a database session or a storage SDK. It needs to read order history and write an export. Python lets us describe those small shapes:

```python title="orders/export.py"
from typing import Protocol

class OrderHistory(Protocol):
    def for_customer(self, customer_id: int) -> list[OrderRow]: ...

class ExportDestination(Protocol):
    def write(self, content: bytes) -> str: ...
```

The constructor can now say exactly what the operation uses:

```python title="orders/export.py"
class OrderExporter:
    def __init__(self, history: OrderHistory, destination: ExportDestination) -> None:
        self._history = history
        self._destination = destination

    def __call__(self, request: ExportOrders) -> ExportResult:
        rows = self._history.for_customer(request.customer_id)
        content = render(rows, request.fmt)
        url = self._destination.write(content)
        return ExportResult(url=url, rows=len(rows))
```

Startup chooses the real implementations:

```python title="bootstrap.py"
order_exporter = OrderExporter(
    history=StoredOrderHistory(worker_database),
    destination=DownloadFiles(storage_client),
)
```

A test can supply small in-memory objects. A web process can supply an `OrderHistory` built around its database connection. The worker can supply another with the lifetime it needs. A new storage provider requires a new `ExportDestination`, while the export rules stay untouched.

The `Protocol` classes do not perform any runtime work. They record the narrow agreement between the export and the objects supplied to it. The freedom comes from the export using only that agreement and from startup choosing the real objects.

The two changes together give the export two kinds of freedom:

- its plain input keeps Flask and the queue outside the operation, while supplied dependencies let each process use objects with the right lifetime;
- its database and storage products can change by supplying new objects that meet the same small descriptions.

The two improvements reinforce each other, but they solve different problems.

## The unusual result

The same approach can be applied to another operation when it starts resisting change.

Placing an order uses inventory, payment, order storage, and notifications. Creating an invoice uses order data and a renderer. Refunding an order has its own input and rules.

<SplitDiagram />

Code that uses the same things and changes together remains together. Code that only shared the word "order" no longer has to travel as one unit.

After a few operations have been treated this way, the application contains classes like these:

```python
place_order = PlaceOrder(orders, payments, inventory, notifications)
refund_order = RefundOrder(orders, payments)
create_invoice = CreateInvoice(orders, renderer)
order_exporter = OrderExporter(history, destination)
```

Each class may have only one public method.

Without the path that led here, that looks like an elaborate way to write a function. With the path in view, the shape is less mysterious.

The operation needed its own home so it could move without unrelated code. Its outside dependencies needed to be supplied so their implementations could change. A class stores those supplied objects, and one public method performs the operation. The method count is a consequence of those choices, not a goal.

Each object represents one thing the application promises it can do: place an order, refund an order, create an invoice, export order history. Frameworks turn outside input into those operations. Providers do general-purpose work on their behalf. The operation holds the decisions unique to this shop.

The application is now easier to change, but it is also more explicit. There are more files, classes, and objects than before.

That cost is real.

## There are rather a lot of objects

The original route imported one module. A later version received one large `OrderService`. Now the application starts several focused objects:

```python title="bootstrap.py"
place_order = PlaceOrder(...)
refund_order = RefundOrder(...)
create_invoice = CreateInvoice(...)
order_exporter = OrderExporter(...)
monthly_statement = CreateMonthlyStatement(...)
# and more as the application grows
```

The HTTP layer may need four of them. A command-line tool needs three. Workers and scheduled jobs have their own lists.

Passing each object directly is a good design. It is visible and easy to follow. If the lists remain short, this can be the end of the story.

When the same wiring starts appearing at every doorway, a small dictionary can put the lookup in one place:

```python title="dispatch.py"
operations = {
    PlaceOrder: place_order,
    RefundOrder: refund_order,
    CreateInvoice: create_invoice,
    ExportOrders: order_exporter,
}

def send(request):
    operation = operations[type(request)]
    return operation(request)
```

The objects still exist. Startup still constructs them. The dictionary changes what callers need to hold.

A route can receive `send` and an input type rather than every operation it might invoke:

```python title="app.py"
@app.post("/orders/export")
def export_orders_route():
    result = send(ExportOrders(customer_id=g.current_user.id))
    return jsonify(url=result.url), 202
```

The worker uses the same entrance:

```python title="worker.py"
def handle_message(message):
    send(ExportOrders(**message.body))
```

<SeamFlowDiagram />

This gives every doorway one way to talk to the application: create a description of the work, send it, receive the result.

The registered input types also form a useful list of the operations available through that entrance. The dictionary knows which object performs each one. The result still appears in the operation's return annotation, although the plain `send()` function cannot preserve that relationship for the type checker.

This small dispatcher is a mediator. Its contribution is limited and useful: callers no longer need to locate or hold each operation object individually.

For many applications, the dictionary is enough.

## Making the dictionary dependable

The first rough edge appears in the editor. A human can see that `ExportOrders` produces `ExportResult`, but the type checker sees a dictionary of different callables. `send()` returns `Any`, so autocomplete disappears after the call.

A missing entry becomes a `KeyError` when that path runs. A callable with the wrong parameter type may sit unnoticed until its first call. Accidentally registering two objects for the same input needs another check.

Those are ordinary problems in a small hand-written dispatcher. Solving them well takes more code than the original dictionary.

[PyMediate](/docs) provides that code.

The input declares its result type:

```python
from dataclasses import dataclass
from pymediate.sync import Request

@dataclass
class ExportResult:
    url: str
    rows: int

@dataclass
class ExportOrders(Request[ExportResult]):
    customer_id: int
    fmt: str = "csv"
```

The exporter declares which input it accepts:

```python
from pymediate.sync import RequestHandler

class OrderExporter(RequestHandler[ExportOrders]):
    def __init__(self, history: OrderHistory, destination: ExportDestination) -> None:
        self._history = history
        self._destination = destination

    def __call__(self, request: ExportOrders) -> ExportResult:
        rows = self._history.for_customer(request.customer_id)
        content = render(rows, request.fmt)
        url = self._destination.write(content)
        return ExportResult(url=url, rows=len(rows))
```

Startup builds the exporter as before and gives it to PyMediate:

```python title="bootstrap.py"
from pymediate.sync import Mediator, Services

order_exporter = OrderExporter(
    history=StoredOrderHistory(worker_database),
    destination=DownloadFiles(storage_client),
)

services = Services().add(order_exporter)
mediator = Mediator(services.provider())
```

Now `mediator.send(ExportOrders(...))` is known to return `ExportResult`. PyMediate checks that the exporter accepts the input it claims to accept. Sending an unregistered request produces an error that names the request.

Underneath the package names is the dictionary we already understood: an input goes in, one callable object receives it, and a result comes back.

PyMediate preserves the result type through `send()`, validates the relationship between an input and its handler, and reports missing or duplicate handlers clearly. It also supplies async and sync variants for applications that need them.

The set of business operations and their boundaries stays the same. PyMediate's scope is the common entrance, making it a little more pleasant to use and maintain.

## The trade

The additional structure has a price.

Reading a route no longer reveals the exporter's filename; you follow the request type to the class that handles it. Describing operations as data introduces input classes. PyMediate adds its `Request`, `RequestHandler`, and `Mediator` types.

Dispatch also takes time. A bare request adds about **a microsecond** to a direct call. [The benchmark](/docs/comparison#what-does-send-cost) is public and reproducible. A database query will dwarf that cost. A tight loop of trivial in-memory work will not.

The trade begins to make sense when the application has enough operations or doorways that the common input-and-result model is easier to understand than lists of directly supplied objects.

If direct calls remain clear, keep them. If the dictionary remains small and its missing type information causes no trouble, keep the dictionary. PyMediate is useful when maintaining that plumbing has become work you would rather not own.

## One change at a time

None of this requires a new project layout or a rewrite.

Start with a change the code is resisting today.

If one operation cannot move without dragging unrelated code with it, give that operation its own home and plain input. If a provider or database choice keeps reaching into the business rules, let the operation describe the small piece it needs and choose the real object at startup.

Apply the step that answers the pressure in front of you, then stop and see whether the change became easier.

Over time, the application may reveal a set of focused operations. Direct calls may remain enough. A dictionary may make their callers simpler. PyMediate may make the dictionary easier to trust.

The codebase did not become difficult through one bad decision, and it does not become approachable through one pattern. It changes one boundary at a time, until an engineer can pick up a two-day improvement and find that it still takes two days.
