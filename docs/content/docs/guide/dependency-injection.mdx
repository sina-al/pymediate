PyMediate ships optional support for [`dependency-injector`](https://python-dependency-injector.ets-labs.org/), so handlers get their dependencies from a DI container instead of being constructed by hand.

```bash
pip install 'pymediate[di]'
```

See [installation](/docs/getting-started/installation) for other package managers, and [troubleshooting](/docs/advanced/troubleshooting#dependencyinjectorserviceprovider-not-available) if the import fails.

## Basic setup

`DependencyInjectorServiceProvider` wraps a `dependency-injector` container and implements PyMediate's `ServiceProvider` [protocol](https://docs.python.org/3/library/typing.html#typing.Protocol), so it drops in wherever `Services.provider()` would go. It indexes handlers by their declared output type — provider names in the container are irrelevant, and providers are not called during indexing:

```python
from dependency_injector import containers, providers
from pymediate import RequestHandler, Mediator
from pymediate.providers import DependencyInjectorServiceProvider

class AppContainer(containers.DeclarativeContainer):
    database = providers.Singleton(Database)

    # The provider name can be anything — PyMediate matches by type, not by name
    user_service = providers.Factory(CreateUserHandler, database=database)
```

```python
container = AppContainer()
provider = DependencyInjectorServiceProvider(container)
mediator = Mediator(provider)

response = await mediator.send(CreateUserRequest(username="alice", email="alice@example.com"))
```

## Modular containers

Child [`providers.Container`](https://python-dependency-injector.ets-labs.org/examples/application-multiple-containers.html) providers are discovered recursively. Their providers appear at the child's position in the root declaration order, which keeps pipeline behavior ordering predictable:

```python
class OrdersContainer(containers.DeclarativeContainer):
    database = providers.Dependency()
    create_order = providers.Factory(CreateOrderHandler, database=database)

class ApplicationContainer(containers.DeclarativeContainer):
    database = providers.Singleton(PostgresDatabase)
    logging = providers.Factory(LoggingBehavior)
    orders = providers.Container(OrdersContainer, database=database)
```

Only declared child containers are descended into. Dependencies injected into a factory are not independently traversed and registered.

## Compose the mediator in the container

Provider indexing does not construct services, so Dependency Injector's documented [`providers.Self()`](https://python-dependency-injector.ets-labs.org/providers/inject_self.html) pattern is safe:

```python
class ApplicationContainer(containers.DeclarativeContainer):
    __self__ = providers.Self()

    create_user = providers.Factory(CreateUserHandler)
    services = providers.Singleton(DependencyInjectorServiceProvider, __self__)
    mediator = providers.Singleton(Mediator, services=services)

mediator = ApplicationContainer().mediator()
```

Constructing the mediator outside the container remains equally valid.

## Opaque factories

Class-backed providers are inferred directly. A function-backed provider needs a concrete return annotation:

```python
def build_handler(database: Database) -> CreateUserHandler:
    return CreateUserHandler(database)

create_user = providers.Factory(build_handler, database=database)
```

A provider whose output type cannot be determined this way — an unannotated factory, a `Selector`, a `Resource` — is rejected at construction time with a `TypeError` naming the provider, rather than being called to discover its type.

The provider graph and inferred types are indexed once. Rebuild the service provider after adding providers or applying an override that changes a provider's output type. Same-type overrides continue to work.

## Factory vs. Singleton providers

Both provider types work — pick the one matching the handler's intended lifetime:

```python
class AppContainer(containers.DeclarativeContainer):
    database = providers.Singleton(Database)

    # Factory: a new handler instance on every resolution
    create_user_handler = providers.Factory(CreateUserHandler, database=database)

    # Singleton: one shared handler instance for the whole application
    metrics_handler = providers.Singleton(RecordMetricsHandler, database=database)
```

Most handlers should be `Factory` unless they're genuinely stateless and safe to share. [Pipeline behaviors registered through the container](/docs/guide/pipeline-behaviors#di-container-integration) respect their scopes the same way, including `ContextLocalSingleton` for per-request lifetimes.

## Testing with a container

Override providers exactly as you would in any `dependency-injector` test:

```python
@pytest.mark.asyncio
async def test_create_user_with_container():
    container = AppContainer()
    container.database.override(providers.Singleton(InMemoryDatabase))

    mediator = Mediator(DependencyInjectorServiceProvider(container))
    response = await mediator.send(CreateUserRequest(username="alice", email="alice@example.com"))

    assert response.username == "alice"
```

If a test doesn't need the container at all, construct the handler directly — see [testing without frameworks](/docs/guide/requests-responses#testing-without-frameworks).
