The mediator is PyMediate's central dispatch point, with two verbs. `send()` receives a request, resolves the right handler through a service provider, executes it (wrapped in any applicable pipeline behaviors), and returns the typed response. `publish()` delivers an event to every subscribed handler and returns nothing.

```python
from pymediate import Mediator, Services

services = Services()
services.add(CreateUserHandler(database))

mediator = Mediator(services.provider())

response = await mediator.send(CreateUserRequest(username="alice", email="alice@example.com"))
```

## What `send()` actually does

1. The application sends a request to the mediator.
2. The mediator looks up the handler **class** registered for this request type — recorded automatically when the `RequestHandler[RequestT]` subclass was defined.
3. It resolves a handler **instance** of that class from the `ServiceProvider`.
4. It discovers every registered `PipelineBehavior` that applies to this request, and either calls the handler directly (no applicable behaviors) or wraps it in a `Pipeline` and calls that.

Step 4 is a built-in fast path: if no behavior applies, the handler is called directly with zero pipeline-construction overhead. See [pipeline behaviors](/docs/guide/pipeline-behaviors) for how behaviors are matched and ordered.

With multiple handlers registered, routing is purely by request type:

```python
services = Services()
services.add(CreateUserHandler(database))
services.add(GetUserHandler(database))
services.add(DeleteUserHandler(database))

mediator = Mediator(services.provider())

create_response = await mediator.send(CreateUserRequest(...))  # → CreateUserHandler
get_response = await mediator.send(GetUserRequest(...))        # → GetUserHandler
delete_response = await mediator.send(DeleteUserRequest(...))  # → DeleteUserHandler
```

## Type safety

`send()` infers its return type from the request's `Request[ResponseT]` parameter:

```python
response = await mediator.send(CreateUserRequest(...))
# response: CreateUserResponse — inferred, no annotation needed

print(response.user_id)   # autocompletes
print(response.invalid)   # mypy error
```

## Publishing events

The mediator's second verb is `publish()`. Where `send()` routes a request to its one handler and returns the response, `publish()` delivers an event to *every* `EventHandler` subscribed to its type — zero, one, or many — and returns nothing:

```python
await mediator.publish(OrderPlaced(order_id=42, customer_email="a@example.com"))
```

Handlers run concurrently via `asyncio.gather`; a raising handler doesn't stop the others, and failures aggregate into an `ExceptionGroup` once every handler has run. Pipeline behaviors wrap `send()` only — publishes go straight to the subscribers. The full semantics, patterns, and error handling are in the [events guide](/docs/guide/events).

## The sync mediator

Sync handlers use a separate mediator: `pymediate.sync.Mediator`, paired with `pymediate.sync.RequestHandler`. The sync API mirrors the async one structurally, but the two don't mix at runtime — a sync `Mediator` never awaits anything, so it can't dispatch an `async def __call__`.

```python
from pymediate.sync import Request, RequestHandler, Mediator, Services

class FetchDataHandler(RequestHandler[FetchDataRequest]):
    def __call__(self, request: FetchDataRequest) -> FetchDataResponse:
        data = self.http_client.get(request.url)
        return FetchDataResponse(data=data)

services = Services()
services.add(FetchDataHandler(http_client))
mediator = Mediator(services.provider())

response = mediator.send(FetchDataRequest(url="https://api.example.com/data"))
```

An application can run both variants side by side — an async `Mediator` for async handlers and a sync one for sync handlers — but each request goes through the mediator that matches its handler. There is no single mediator that routes to either kind.

The same split applies to events: `pymediate.sync.Mediator.publish()` dispatches to `pymediate.sync.EventHandler` subscribers — sequentially in registration order, unlike the async mediator's [concurrent delivery](/docs/guide/events#sync-events).

## Error behavior

If no handler is registered for a request type, `send()` raises `HandlerNotFoundError`:

```python
from pymediate import HandlerNotFoundError

try:
    response = await mediator.send(UnregisteredRequest())
except HandlerNotFoundError as e:
    print(f"No handler for request: {e}")
```

Exceptions raised *inside* handlers propagate unchanged — the mediator never catches or wraps them:

```python
class CreateUserHandler(RequestHandler[CreateUserRequest]):
    async def __call__(self, request: CreateUserRequest) -> CreateUserResponse:
        if not request.email:
            raise ValueError("Email is required")
        ...

try:
    await mediator.send(CreateUserRequest(username="test", email=""))
except ValueError as e:
    print(f"Validation error: {e}")
```

See [error handling](/docs/guide/error-handling) for keeping domain errors independent of the framework at the edge of your application.

## Compose behaviors, don't subclass

`Mediator` isn't designed to be subclassed — there's no protected hook, and `send()`'s behavior isn't meant to vary between instances. For logging, timing, validation, or any other cross-cutting concern, register a [pipeline behavior](/docs/guide/pipeline-behaviors). Behaviors are auto-discovered from the same `Services`/`ServiceProvider` as handlers:

```python
from pymediate import Request, PipelineBehavior, Services, Mediator

class LoggingBehavior(PipelineBehavior[Request]):
    async def __call__(self, request, next):
        print(f"Before: {type(request).__name__}")
        response = await next()
        print(f"After: {type(request).__name__}")
        return response

class TimingBehavior(PipelineBehavior[Request]):
    async def __call__(self, request, next):
        start = time.time()
        response = await next()
        print(f"Took {time.time() - start:.3f}s")
        return response

services = Services()
services.add(LoggingBehavior())   # outermost
services.add(TimingBehavior())    # inner
services.add(CreateUserHandler())

mediator = Mediator(services.provider())
# Every request runs: Logging → Timing → RequestHandler
```

Registration order sets execution order, outermost first:

```
Request
  → Logging (outermost)
    → Timing
      → RequestHandler
      ← response
    ← Timing
  ← Logging
```

This gets you everything a subclass would — shared setup, teardown, error handling — while staying independently testable and reusable.

## Testing with the mediator

Unit-test handlers directly, without a mediator at all. Use a real mediator for integration tests of the full flow:

```python
@pytest.mark.asyncio
async def test_with_mediator():
    services = Services()
    services.add(CreateUserHandler(mock_db))
    mediator = Mediator(services.provider())

    response = await mediator.send(CreateUserRequest(username="test", email="test@example.com"))

    assert response.user_id > 0
```

For consumers that *send* requests or *publish* events, fake the mediator — it's two methods:

```python
class MockMediator:
    def __init__(self):
        self.sent_requests = []
        self.published_events = []

    async def send(self, request):
        self.sent_requests.append(request)
        if isinstance(request, CreateUserRequest):
            return CreateUserResponse(user_id=999, username=request.username)

    async def publish(self, event):
        self.published_events.append(event)

@pytest.mark.asyncio
async def test_place_order_sends_expected_requests():
    mock_mediator = MockMediator()
    handler = PlaceOrderHandler(mediator=mock_mediator, database=mock_db)

    await handler(PlaceOrderRequest(...))

    assert isinstance(mock_mediator.sent_requests[0], ChargePaymentRequest)
```

## Best practices

- **One mediator instance per application.** Build it once at startup; don't construct one per request.
- **Inject the mediator into handlers** that need to dispatch further requests — no globals.
- **Don't mix dispatch styles.** If a handler composes through the mediator, don't also hand it direct references to other handlers.
- **Swap providers, not call sites.** `ServiceProvider` is a protocol; you can move from `Services` to a [dependency-injector container](/docs/guide/dependency-injection) without changing any `mediator.send()` call.
