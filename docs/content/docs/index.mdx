PyMediate is a small library for Python 3.12+ that gives your application one type-safe way to ask for things. You describe what you want as a plain object, hand it to a single dispatcher, and get back a fully typed response:

```python
response = await mediator.send(CreateUser(username="alice", email="alice@example.com"))
# response is UserCreated — inferred from the request type, checked by mypy
```

That one line is most of the library. The rest of this page is what it's for and how the pieces fit.

## The problem

In every application there's code that *wants* something done and code that *knows how* to do it. Connecting the two with a direct call is the right way to start — and as an application grows, it's also how call sites quietly accumulate knowledge they never needed: the concrete class, its constructor, its dependencies, its framework context. Knowledge is coupling, and coupling compounds. One day a simple request — *run this operation in a background job, add a CLI for that feature* — turns out to require a cascade of other changes first, and the codebase has become the kind nobody wants to touch.

## The mediator pattern

The [mediator pattern](https://en.wikipedia.org/wiki/Mediator_pattern) removes the direct call. Instead of invoking business logic, you describe what you want as a plain message — a **request** — and hand it to a single dispatcher — the **mediator** — which routes it to the one place that knows how to do it — a **handler**:

```python
from dataclasses import dataclass
from pymediate import Request, RequestHandler

@dataclass
class CreateUser(Request[UserCreated]):   # the message, declaring its response type
    username: str
    email: str

class CreateUserHandler(RequestHandler[CreateUser]):   # the one place it's handled
    async def __call__(self, request: CreateUser) -> UserCreated:
        ...

user = await mediator.send(CreateUser(username="alice", email="alice@example.com"))
```

Now the caller knows only the message it sends and the mediator it sends it to. The handler knows only the request it receives. The web view, the CLI command, and the background job all send the same `CreateUser` and none of them can tell — or need to know — how it's fulfilled. And cross-cutting concerns get a real home: a **pipeline behavior** wraps every dispatch, so logging or validation is written once instead of scattered across handlers.

Requests are one-to-one: something is *wanted*, and exactly one handler does it. The same mediator also covers the one-to-many case — something *happened*, and any number of places care. `mediator.publish()` delivers an **event** to every subscribed **event handler**, with no response and no knowledge of who's listening:

```python
await mediator.publish(UserRegistered(user_id=user.user_id))
# the welcome email, the analytics record, the audit log — each a separate EventHandler
```

## Why PyMediate?

PyMediate implements the pattern entirely inside Python's type system — built on [PEP 695](https://peps.python.org/pep-0695/) generics, new in Python 3.12 — so the indirection costs you nothing you'd have with a direct call:

- **The link is a type.** `RequestHandler[CreateUser]` declares what a handler handles, and dispatch is keyed on the class itself. Rename the request and every handler follows in the refactor.
- **You always know what comes back.** A request inherits `Request[UserCreated]`, and from then on `mediator.send()` returns `UserCreated` everywhere — inferred by mypy, autocompleted by your editor, no casts. The response can be any type at all: no framework base class, no serialization methods.
- **Wiring mistakes surface before requests flow.** RequestHandler signatures are validated the moment the class is defined — a wrong parameter type, a missing return type, a sync/async mixup all raise at import. A second handler for the same request raises immediately, pointing at the file and line of the first.
- **Sync and async are equals.** The top-level API is async; `pymediate.sync` mirrors it class for class — switch the import, drop the `await`, done.
- **It travels light.** Zero runtime dependencies, ships `py.typed`, and is checked under `mypy --strict`. `dependency-injector` support is an optional extra.

Evaluating it against other libraries? [How it compares](/docs/comparison) has a feature-by-feature look at the Python mediator ecosystem and a reproducible benchmark of what `send()` and `publish()` cost over direct calls.

## The moving parts

| Concept | Role |
| --- | --- |
| `Request[T]` | A message declaring what it wants and the response type `T` it expects |
| `RequestHandler[R]` | The single place where a request type `R` gets handled |
| `Event` | A fact that already happened, published to whoever subscribed — no response |
| `EventHandler[E]` | One subscriber's reaction to an event type `E`; any number may subscribe |
| `Mediator` | Routes each request to its handler and each event to its subscribers; the only thing callers depend on |
| `PipelineBehavior` | Optional middleware wrapping handlers — logging, validation, transactions |
| `Services` / `ServiceProvider` | Registration and resolution of handlers, behaviors, and their dependencies |
