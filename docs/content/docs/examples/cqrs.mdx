title: CQRS pattern
description: Separating the command (write) and query (read) sides of a feature with CQRS.
---

Command Query Responsibility Segregation (CQRS) separates operations that change state
(commands) from operations that read state (queries). PyMediate uses the same `Request` and
`Mediator.send()` API for both. The separation comes from their handlers, models, and stores,
not from a second dispatch API or naming alone. See
[requests and responses: CQRS](/docs/guide/requests-responses#cqrs-commands-and-queries) for the
underlying concepts.

This page uses a product catalog with a transactional online transaction processing (OLTP)
write store and a read-optimized online analytical processing (OLAP) store. The inline version
uses in-memory data so it can run as one snippet. The
[`130-cqrs` example](https://github.com/sina-al/pymediate/tree/main/examples/130-cqrs) uses SQLite,
DuckDB, a transactional outbox, and a projection worker. See
[Keeping the read side in sync](#keeping-the-read-side-in-sync).

```python
import asyncio
from dataclasses import dataclass, field
from pymediate import Request, RequestHandler, Mediator, Services

@dataclass
class Product:
    product_id: int
    name: str
    price: float
    stock: int

class ProductWriteStore:
    """Simulates a write-optimized primary database."""
    def __init__(self):
        self._products: dict[int, Product] = {}
        self._next_id = 1

    def create(self, name: str, price: float, stock: int) -> Product:
        product = Product(product_id=self._next_id, name=name, price=price, stock=stock)
        self._products[product.product_id] = product
        self._next_id += 1
        return product

    def adjust_stock(self, product_id: int, delta: int) -> Product:
        product = self._products[product_id]
        product.stock += delta
        return product

class ProductReadStore:
    """Simulates a read-optimized replica, kept in sync separately.

    Reads straight from the write store here only to stay runnable in one block. In the
    130-cqrs example this is a separate DuckDB table, fed by a projection worker.
    """
    def __init__(self, write_store: ProductWriteStore):
        self._write_store = write_store

    def find(self, product_id: int) -> Product | None:
        return self._write_store._products.get(product_id)

    def search(self, min_price: float = 0) -> list[Product]:
        return [p for p in self._write_store._products.values() if p.price >= min_price]
```

## Commands

Commands change state and return only what the caller needs to proceed — an ID, a new count, a status:

```python
@dataclass
class CreateProductResponse:
    product_id: int

@dataclass
class CreateProductCommand(Request[CreateProductResponse]):
    name: str
    price: float
    stock: int = 0

class CreateProductHandler(RequestHandler[CreateProductCommand]):
    def __init__(self, store: ProductWriteStore):
        self.store = store

    async def __call__(self, request: CreateProductCommand) -> CreateProductResponse:
        product = self.store.create(request.name, request.price, request.stock)
        return CreateProductResponse(product_id=product.product_id)

@dataclass
class AdjustStockResponse:
    product_id: int
    new_stock: int

@dataclass
class AdjustStockCommand(Request[AdjustStockResponse]):
    product_id: int
    delta: int

class AdjustStockHandler(RequestHandler[AdjustStockCommand]):
    def __init__(self, store: ProductWriteStore):
        self.store = store

    async def __call__(self, request: AdjustStockCommand) -> AdjustStockResponse:
        product = self.store.adjust_stock(request.product_id, request.delta)
        return AdjustStockResponse(product_id=product.product_id, new_stock=product.stock)
```

## Queries

Queries do not mutate state. They return a read-specific projection containing the fields the
caller needs:

```python
@dataclass
class GetProductResponse:
    product_id: int
    name: str
    price: float
    stock: int

@dataclass
class GetProductQuery(Request[GetProductResponse]):
    product_id: int

class GetProductHandler(RequestHandler[GetProductQuery]):
    def __init__(self, store: ProductReadStore):
        self.store = store

    async def __call__(self, request: GetProductQuery) -> GetProductResponse:
        product = self.store.find(request.product_id)
        if product is None:
            raise ValueError(f"Product {request.product_id} not found")
        return GetProductResponse(**vars(product))

@dataclass
class SearchProductsResponse:
    products: list[Product] = field(default_factory=list)

@dataclass
class SearchProductsQuery(Request[SearchProductsResponse]):
    min_price: float = 0

class SearchProductsHandler(RequestHandler[SearchProductsQuery]):
    def __init__(self, store: ProductReadStore):
        self.store = store

    async def __call__(self, request: SearchProductsQuery) -> SearchProductsResponse:
        return SearchProductsResponse(products=self.store.search(request.min_price))
```

## Wiring it together

Both sides register on the same `Services` collection and go through the same `Mediator` — CQRS is a split in your domain model, not in PyMediate's dispatch mechanism:

```python
write_store = ProductWriteStore()
read_store = ProductReadStore(write_store)

services = Services()
services.add(CreateProductHandler(write_store))
services.add(AdjustStockHandler(write_store))
services.add(GetProductHandler(read_store))
services.add(SearchProductsHandler(read_store))
mediator = Mediator(services.provider())

async def main():
    created = await mediator.send(CreateProductCommand(name="Keyboard", price=49.99, stock=10))
    print(created)
    # Output: CreateProductResponse(product_id=1)

    adjusted = await mediator.send(AdjustStockCommand(product_id=created.product_id, delta=-1))
    print(adjusted)
    # Output: AdjustStockResponse(product_id=1, new_stock=9)

    fetched = await mediator.send(GetProductQuery(product_id=created.product_id))
    print(fetched)
    # Output: GetProductResponse(product_id=1, name='Keyboard', price=49.99, stock=9)

    results = await mediator.send(SearchProductsQuery(min_price=10))
    print(results)
    # Output: SearchProductsResponse(products=[Product(product_id=1, name='Keyboard', price=49.99, stock=9)])

asyncio.run(main())
```

## Keeping the read side in sync

The snippet above lets the read store access the write store's data so it can run in one block.
A separate read store needs a durable way to receive each committed change. Writing both stores
inside the command handler, or projecting in an inline event subscriber during `send()`, creates
a dual write: the write store can commit while the read-store update fails, with no durable retry
record.

The [`130-cqrs` example](https://github.com/sina-al/pymediate/tree/main/examples/130-cqrs) uses
a transactional outbox and projection worker:

- the command writes the domain row **and** an event row to a **transactional outbox** in one
  SQLite transaction, so both commit or both roll back;
- a background **projection worker** reads the outbox in order and applies batches to the DuckDB
  read model under a **checkpoint**, so a crash can't leave the read model ahead of or behind
  its recorded position;
- the command publishes an `OutboxAppended` event to wake the worker. The outbox is the durable
  record, and polling lets the worker recover if the in-memory event is lost.

The read side is then eventually consistent, which the example makes explicit: commands return
an outbox position, and a caller can wait until the read model's checkpoint reaches it. The
example's `docs/architecture.md` walks through the recovery cases and the real-world analogues
(Postgres/Turso for the write store, Snowflake/ClickHouse for the read model, Debezium or Kafka
Connect for the worker).

<Callout type="info">
  The mediator dispatches commands and queries through the same `send()` method. The handlers,
  stores, outbox, and projection worker define the CQRS separation.
</Callout>
